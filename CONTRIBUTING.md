# Contributing

Contributions are welcome if they make `constructs.md` more useful for real engineering work with AI coding agents.

Good contributions:

- clearer construct templates
- better agent prompts
- stronger examples
- real anti-patterns
- review checklists
- small wording improvements

Weak contributions:

- turning this into a heavyweight methodology
- adding generic architecture fluff
- adding vendor-specific marketing
- bloating the template with implementation details

## Contribution rule

Every change should preserve the core idea:

> A construct is a mental model for product/system responsibility, not a folder map.

## Pull request checklist

- [ ] Does this make the repo more practical?
- [ ] Does this help agents preserve architecture boundaries?
- [ ] Is the language concrete rather than abstract?
- [ ] Did you avoid adding unnecessary process?
